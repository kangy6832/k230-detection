## 目标

在 K230 平台上实现一个最小可用的拍照应用，能够打开摄像头、获取并处理视频流，并将拍摄的照片保存到 [photos/](../photos/)。

## 执行计划

### 1. 需求确认与环境摸底
- 继续确认 K230 目标环境的运行方式、可用摄像头接口、图像编码格式以及照片保存路径要求。
- 补充项目入口形式，确认是单文件脚本、模块化 Python 程序，还是其他启动方式。
- 如发现需求或环境信息不足，先记录到 [questions.md](questions.md) 再继续推进。

### 2. 项目结构设计
- 保持当前目录结构不变，只在需要时补充最少量文件。
- 明确应用入口、配置文件和输出目录的职责边界。
- 约定摄像头采集、视频处理、拍照保存三个职责尽量分离，避免把所有逻辑堆在入口文件里。

### 3. 摄像头采集链路实现
- 编写打开摄像头的初始化流程。
- 实现视频流读取主循环，确保可以持续获取帧数据。
- 增加最基础的视频处理步骤，至少保留可扩展的处理接口，方便后续加入检测、裁剪或缩放等逻辑。

### 4. 拍照与存储实现
- 增加拍照触发逻辑，支持在指定时机保存当前帧。
- 将图片写入 [photos/](../photos/) 目录，文件命名需要稳定且避免覆盖。
- 保存失败时输出明确错误信息，并保留现场上下文，方便排查设备权限、路径或格式问题。

### 5. 本地验证
- 如果新增或修改 Python 入口脚本，先执行 `python3 -m py_compile` 检查语法。
- 若存在可运行的最小验证流程，再确认摄像头初始化、取帧和保存图片三个关键动作是否都能完成。
- 记录验证结果与异常现象，便于后续修复。

### 6. 交付整理
- 更新 [progress.md](progress.md) 记录完成内容和遗留问题。
- 在 [summsry.md](summsry.md) 中补充简短总结，说明最终实现和注意事项。
- 如后续增加了新的工作流约束，再同步回 [AGENTS.md](../AGENTS.md)。

## 风险与待确认事项

- K230 上具体可用的摄像头 API、驱动和运行环境尚未确认。
- 当前没有发现项目级构建或测试命令，需要在后续实现阶段继续从项目文件中发现。
- `photos/` 的目标保存格式、分辨率和命名规则如果有额外要求，需要先确认再实现。

## 完成标准

- 能成功打开摄像头并持续获取视频流。
- 能对视频流帧进行基本处理或预留处理入口。
- 能将拍摄的照片保存到 [photos/](../photos/)。
- 相关文档已更新，且实现过程中的问题已记录。
