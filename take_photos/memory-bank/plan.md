## 目标

基于 ALIENTEK 的 DNK230D 照片示例，规划一个 CanMV / MicroPython 的拍照应用。应用需要能打开摄像头、持续获取视频流，支持按键和定时两种拍照方式，并将照片保存到 [photos/](../photos/) 中，格式覆盖 JPG 和 BMP。

## 执行计划

### 1. 对齐示例与实现边界
- 以 wiki 示例的技术路线为准，默认使用 `sensor`、`display`、`media`、`image` 和 `machine` 相关接口。
- 以“最小拍照功能”为核心，不额外扩展无关能力，但保留后续扩展视频处理的入口。
- 明确输入输出：摄像头预览、照片保存、文件命名、格式选择、保存目录。

### 2. 项目结构设计
- 保持当前目录结构尽量不变，只在必要时补充最少量文件。
- 入口优先放在仓库根部，便于在 K230 环境中直接运行或拷贝。
- 如果需要配置，单独拆出轻量配置文件，避免把设备参数和业务逻辑混在一起。

### 3. 摄像头与预览链路
- 按示例初始化摄像头，并确认预览通道与拍照通道可分离。
- 预览侧负责持续显示画面，拍照侧负责输出更适合保存的帧。
- 预留最基础的视频处理钩子，后续如果需要做裁剪、缩放或简单图像处理，可以直接接入。

### 4. 拍照触发与格式保存
- 支持按键触发拍照：保留 KEY0 / KEY1 对应不同保存格式的能力。
- 同时支持定时拍照，方便没有按键输入时仍能自动采集照片。
- 统一保存到 [photos/](../photos/) 目录，命名需要避免覆盖并能体现时间或序号。
- JPG / BMP 的保存逻辑分开封装，便于后续增加其他格式或调整压缩质量。

### 5. 运行与验证
- 如果新增或修改 Python 入口脚本，先执行 `python3 -m py_compile` 检查语法。
- 在 K230 / CanMV 环境中验证摄像头初始化、画面显示、按键拍照、定时拍照和文件落盘是否都正常。
- 记录验证中的异常、兼容性差异和硬件限制，避免后续重复踩坑。

### 6. 文档整理
- 更新 [progress.md](progress.md) 记录当前阶段状态和问题。
- 完成后在 [summary.md](summary.md) 中写清实现结论和注意事项。
- 如后续新增工作流约束，再同步到 [AGENTS.md](../AGENTS.md)。

## 风险与待确认事项

- 需要在实现前确认 K230 运行环境里 CanMV / MicroPython 的具体版本和可用接口。
- 需要确认按键触发时是否要求严格对应 KEY0 / KEY1，还是可以提供替代输入方式。
- `photos/` 目录在设备侧和项目侧是否需要同步，当前按项目内保存来规划。
- JPG 保存质量、BMP 是否需要做额外压缩或转换，当前按示例能力规划。

## 完成标准

- 摄像头可以成功初始化并持续显示或获取视频流。
- 支持按键拍照和定时拍照两种触发方式。
- 照片可以保存为 JPG 和 BMP，并写入 [photos/](../photos/)。
- 相关进度、总结和问题记录已更新。
